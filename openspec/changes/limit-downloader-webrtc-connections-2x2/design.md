## Context

目前下載端之間的 WebRTC 傳輸流程已具備背壓控制、CRC 驗證、事件回報與逾時處理，但缺少每個下載端在上傳/下載方向上的連線上限治理；同時，中控分派端在建立下載端互傳配對時，尚未明確以方向性連線數作為前置條件。當同時參與的節點增加時，可能造成單一下載端承擔過多連線，並讓分派結果偏離可承載容量。

本次變更在兩個層面補齊約束：

- 下載端：新增方向性上限（upload/download 各 2）。
- 中控端：分派前檢查來源 `upload < 2`、目標 `download < 2`，且目標缺少來源持有片段，才允許建立互傳配對。

## Goals / Non-Goals

**Goals:**

- 明確定義下載端 WebRTC 連線上限：最多 2 條上傳連線、2 條下載連線。
- 在連線建立前套用方向性容量檢查，達上限時拒絕新增同向連線。
- 中控分派端在指派下載端互傳前，必須以前述方向性連線容量與缺片條件過濾候選配對。
- 保持既有傳輸協定事件語義不變，避免影響目前系統整體邏輯。

**Non-Goals:**

- 不改變稀缺優先（rarest-first）與進度平衡等既有排序原則。
- 不調整 DataChannel 封包切片、背壓與 CRC 驗證機制。
- 不新增通訊訊息型別或改動外部 API。

## Decisions

1. 採用「方向分離」的固定容量模型（upload/download 各自上限 2）

- 決策：在下載端維護上傳與下載活躍連線數，分別以常數上限判斷是否可建立新連線。
- 原因：需求明確指定 2 上傳、2 下載；方向分離可避免單一方向耗盡全部名額。
- 替代方案：
  - 共享總額上限（例如總共 4 條）: 會導致單向壅塞時影響另一方向可用性，不符合需求語意。
  - 動態上限（依頻寬自適應）: 複雜度高且超出本次範圍。

2. 將容量檢查放在「連線建立入口」而非傳輸中途

- 決策：在建立 PeerConnection / 配置傳輸角色前先檢查並決定是否允許。
- 原因：可最小化對既有傳輸流程的干擾，且達到「其餘邏輯不變」要求。
- 替代方案：
  - 建立後再淘汰超額連線: 會增加震盪與額外關閉成本。

3. 連線釋放事件驅動可用名額回收

- 決策：在正常完成、失敗、逾時與主動關閉時統一回收對應方向的連線名額。
- 原因：避免資源計數洩漏造成長時間不可建立新連線。
- 替代方案：
  - 週期性重算名額: 實作更複雜且回收不即時。

4. 中控分派採「容量與缺片」雙條件過濾

- 決策：中控在指派下載端互傳時，僅挑選「來源上傳連線數 < 2、目標下載連線數 < 2、且目標缺少來源片段」的配對。
- 原因：可將端點端限制前移到排程決策，降低無效分派與重試。
- 替代方案：
  - 沿用舊規則（僅判斷是否有缺片）: 可能產生超額候選，增加拒絕或失敗成本。
  - 只檢查單邊容量: 無法完整避免來源或目標端任一方向滿載。

## Risks / Trade-offs

- [風險] 連線結束路徑若遺漏名額回收，可能造成假性滿載 → [緩解] 統一連線生命週期收斂點，並對每種結束事件加上測試。
- [風險] 中控持有的端點連線狀態若延遲，可能誤判容量導致分派失準 → [緩解] 以最新事件回報更新狀態，分派前再次驗證候選條件。
- [風險] 上限固定為 2/2 在高頻寬環境下可能不是最佳吞吐 → [緩解] 先滿足穩定性需求，後續可在新變更中引入可配置化。
- [取捨] 拒絕新連線可提升穩定度，但在短期內可能降低峰值併發傳輸量。

## Migration Plan

1. 在下載端連線管理模組加入方向性上限常數與檢查函式。
2. 將檢查接入既有連線建立流程。
3. 在中控配對邏輯加入分派前雙條件檢查（來源 upload < 2、目標 download < 2、且存在缺片）。
4. 在連線結束事件加入名額回收，涵蓋成功與錯誤路徑。
5. 以既有測試流程驗證：上限與分派條件生效且既有傳輸功能無回歸。

Rollback Strategy:

- 若上線後發生非預期阻塞，可回退此變更並恢復原本無上限的連線建立策略。
