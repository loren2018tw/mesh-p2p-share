## Context

目前中控配對主要以「找到缺片端點後，從可用來源中挑一個符合上傳限制的來源端點」為核心，區塊本身則多半依索引順序被分配。這種做法在分享端持有完整檔案、多個下載端同時啟動時，會導致多個端點早期拿到相近的片段集合，削弱後續下載端互相供片的機會。

本次變更不打算放寬既有單端點同時 1 上傳 / 1 下載限制，而是在既有限制下提升整體片段擴散速度與群體吞吐。約束條件包括：不變更現有 WebSocket 訊息型別、不新增額外外部儲存、排程仍由 Rust 中控記憶體內狀態即時計算。

## Goals / Non-Goals

**Goals:**

- 讓分享端在下載初期優先將不同片段分散給不同下載端，提升網狀擴散速度。
- 將中控排程從依序片段分配改為依「片段稀缺性 + 下載端持有量」決定優先順序。
- 在不改變通訊協定的前提下，使排程結果更公平，避免某些端點長時間落後。

**Non-Goals:**

- 不調整單端點同時上傳/下載數量限制。
- 不引入新的資料傳輸協定、持久化排程資料庫或額外服務。
- 不嘗試解決 WebRTC 建連失敗、超時重試策略等與本變更無直接關聯的問題。

## Decisions

### 1. 配對以片段為中心，而非以下載端為中心

現有邏輯是「找到缺片端點 → 替它找來源端」，下一個被掃描到的端點拿到的片段主要取決於掃描順序。新邏輯改為「列出所有片段並依持有者數量升冪排序 → 從最稀缺片段開始找配對」。

具體流程：

1. 遍歷所有端點的 `owned_chunks`，對每個片段計算持有者數量。
2. 依持有者數量升冪排列片段（最稀缺優先）。
3. 對每個片段，找出缺少該片段且下載容量未滿的端點集合（若多個，優先選已持有片段數最少者）。
4. 再找持有該片段且上傳容量未滿的來源端點集合（優先選上傳數最低者）。
5. 若兩個集合都非空，完成一次指派；否則跳至下一個片段繼續。

選擇此方案是因為「哪個片段最難取得」比「哪個下載端最需要服務」更能直接反映系統整體瓶頸；以片段稀缺性驅動分配，能最快讓稀缺片段擴散至更多端點，進而提升整體可交換性。

替代方案：

- 仍以端點為中心，加上稀缺性二次排序：需要兩層排序且結果較難預測，容易在端點間不公平。

### 2. 同一輪已分配的片段不再重複分配

每次掃描時維護一個 `assigned_this_round: HashMap<file_id, HashSet<chunk_index>>` 暫存集合，每完成一次指派就將該片段記入。後續繼續掃描時跳過已分配片段，避免一輪內把同一片段重複指派給多個端點。掃描結束後此集合即丟棄，下一輪重建。

這讓分享端初始分配自然實現分散效果：因為最稀缺的片段通常各不相同，每個下載端在一輪中會被分到不同片段。

### 3. 下載端優先順序：已持有片段數較少者優先

若同一個稀缺片段有多個合格下載端，優先選擇已持有片段數最少的端點，縮小進度差距。同分時再依上傳/下載負載打破平手。

### 4. 排程維持事件驅動，即時計算，不引入持久化佇列

每次配對掃描直接基於最新 `owned_chunks`、`upload_count`、`download_count` 重新計算片段稀缺度，不維護跨事件的排序狀態。此設計讓行為可預測且不需處理佇列失效問題。

## Risks / Trade-offs

- [每次掃描需計算片段稀缺性] → 以單檔案現有端點持有資訊即時計算，先維持簡單；若未來端點數增大再考慮快取。
- [公平性排序可能讓個別端點短期吞吐下降] → 以整體完成時間與片段擴散速度為優先，並保留既有負載限制避免單點過載。
- [規則變多後排程較難除錯] → 在中控日誌中記錄被分配片段、稀缺度與被挑選下載端的持有量，方便驗證決策。
