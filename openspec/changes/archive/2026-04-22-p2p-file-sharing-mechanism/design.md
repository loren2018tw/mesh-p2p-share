## Context

目前的應用程式為 Rust 後端搭配 Vue/Vuetify 前端，但尚未具備分享大型檔案的能力。為了讓區域網路或已知環境中的多個節點能有效率地分享檔案，我們將引入 P2P 傳輸機制。透過分享端作為中控中心進行動態連線調度，並利用 WebRTC 進行端點對端點間的直接資料傳輸，以減少分享端的頻寬壓力。

## Goals / Non-Goals

**Goals:**
- 實作 50MB 為單位的檔案分塊與 CRC32 驗證。
- 實作中控中心動態排程演算法，能根據各端點的上傳連線數進行負載平衡。
- 實作 Web 端的 P2P 下載邏輯，支援隨機請求缺少的區塊以提升交換多樣性。
- 使用 WebRTC 進行區塊資料傳輸，並具備背壓 (Backpressure) 控制機制。
- 整合 File System Access API 處理瀏覽器端的檔案存檔。

**Non-Goals:**
- 複雜的 P2P 拓樸探索（例如 DHT）：本階段依賴 WebSocket 作為集中式信號交換與調度中心。
- 惡意節點防護：目前以 CRC32 防範隨機錯誤，未採用較耗能的 SHA256 來防止惡意偽造。
- NAT 穿透 (TURN Server) 的外部署：假設在可建立連線的環境中，優先依賴本機或 STUN，不強制建置外部中繼伺服器。

## Decisions

1. **檔案切塊策略：50MB 與 CRC32**
   - *Rationale*: 50MB 是權衡訊令開銷（過小導致頻繁請求）與驗證成本（過大導致驗證失敗時重傳成本高）的適中大小。CRC32 計算速度極快，能有效防範傳輸過程中的隨機錯誤。

2. **動態排程與負載平衡**
   - *Rationale*: 中控中心維護所有端點的狀態（擁有的區塊、當前上傳/下載連線數）。當 B 端點請求區塊時，中控中心會從擁有該區塊的節點中，挑選「目前上傳連線數最少」的節點來建議。若所有候選節點的上傳連線數都滿載，則回傳 `等待 3 秒` 的指令，避免單一節點被過度消耗。

3. **隨機區塊請求 (Random Piece Selection)**
   - *Rationale*: 為了讓新加入的下載端能盡快對其他下載端產生貢獻，下載端不按順序下載，而是從自己缺少的區塊清單中隨機挑選。這能確保環境中的區塊多樣性，最大化 P2P 網路的交換效率。

4. **WebRTC 背壓 (Backpressure) 控制**
   - *Rationale*: WebRTC DataChannel 的緩衝區有限。在傳輸 50MB 的資料時，發送端需將資料進一步切分成小塊（如 64KB~256KB），並監控 `bufferedAmount`，只有在緩衝區低於閾值時才繼續寫入，以防止記憶體溢出或連線斷開。

5. **瀏覽器端檔案寫入：File System Access API**
   - *Rationale*: 允許網頁應用程式在使用者授權後，取得特定資料夾與檔案的直接寫入權限。可透過 `FileSystemWritableFileStream` 的 `seek()` 與 `write()` 支援隨機區塊寫入，不必將數 GB 的檔案全部塞在記憶體中。

## Risks / Trade-offs

- **[Risk] File System Access API 瀏覽器相容性限制** → *Mitigation*: 確保在支援的瀏覽器（如 Chrome, Edge）執行。若遇到不支援的環境，可考慮回退 (fallback) 到一般 `Blob` 下載機制，但 P2P 體驗與記憶體使用會大受影響（本專案暫以支援的瀏覽器為主）。
- **[Risk] WebRTC 連線建立失敗或延遲** → *Mitigation*: 中控中心需支援連線失敗回報機制。若 B 無法與 C 建立連線，可回報給中控中心，並重新發起其他區塊或向其他節點請求。
- **[Risk] 單一節點過載** → *Mitigation*: 嚴格執行中控中心的負載平衡機制，並在 WebRTC 傳輸中確實實作背壓控制。
