## Context

本設計文件定義 P2P 檔案分享系統分享端（Rust-app）的實作架構。主要涉及 Tauri 的 Rust 後端與 Vue/Vuetify 前端之間的整合，特別是在網路通訊安全（HTTPS/WebSocket）與動態檔案管理方面的實作細節。

## Goals / Non-Goals

**Goals:**
- 實作具備動態埠號分配能力的 HTTPS Web Server。
- 整合 WebSocket 服務於同一生命週期。
- 提供直觀的檔案分享與移除介面。
- 提供便捷的連線網址複製與 QR-Code 掃描功能。

**Non-Goals:**
- 本次設計不包含身分驗證機制（由自簽署憑證提供基礎安全）。
- 不包含 NAT 穿透實作（假設在同一區域網路或已有中繼）。

## Decisions

### 1. Web 框架與安全傳輸
- **決定**: 使用 `axum` 配合 `axum-server` 與 `rustls` 處理 HTTPS 請求。
- **理由**: `axum` 是 Rust 社群主流框架，具備優秀的非同步處理能力，且與 WebSocket 整合良好。
- **自簽署憑證**: 預計使用 `rcgen` 在運行時生成或載入預生成的憑證檔案。

### 2. 埠號檢測與分配
- **決定**: 啟動時優先嘗試綁定 `0.0.0.0:4343`。若綁定失敗（`AddrInUse`），則綁定 `0.0.0.0:0` 讓作業系統自動分配可用埠號。
- **理由**: 4343 為約定埠號，若衝突則應保證服務能正常啟動，而非報錯終止。

### 3. 視窗行為管理
- **決定**: 在 `src-tauri/src/main.rs` 的 `setup` hook 中，透過 `window.maximize()` 實現啟動最大化。
- **理由**: 符合使用者在大螢幕管理多個分享檔案的操作習慣。

### 4. QR-Code 實作
- **決定**: 前端使用 `qrcode.vue` 套件根據目前的服務網址即時渲染。
- **理由**: 相比於後端生成圖片傳輸，前端生成能減輕後端負擔並提高介面回應速度。

## Risks / Trade-offs

- **[Risk] 自簽署憑證警告** → [Mitigation] 在 UI 提示下載端使用者需接受安全例外或下載憑證。
- **[Risk] 隨機埠號導致難以預期網址** → [Mitigation] 透過 UI 清晰顯示當前網址，並提供一鍵複製與 QR-Code 供下載端快速接入。
